Real-Time Space Weather Analysis & Prediction 🌌
16 years of historical data (320 MB, 500K+ events) — time-series analysis revealing CMEs precede geomagnetic storms by 2–3 days with 70% correlation.
Automated ETL pipeline via Apache Airflow ingesting 500+ daily events from NASA/NOAA APIs with preprocessing & quality checks.
Interactive Streamlit dashboard for real-time monitoring and historical pattern visualization.
PythonMySQLScikit-learn Apache AirflowDockerStreamlitGit
GitHub Live Demo
// how I approached this
the problem

Space weather events are well-documented individually but the relationships between them across time aren't easy to surface. I wanted to ingest real data continuously, store it properly, and find those patterns automatically rather than requiring a researcher to dig through CSVs.

key decisions
Why Airflow over a cron job? NASA/NOAA APIs have rate limits and occasional downtime. Airflow gave me retry logic, task dependency management, and pipeline visibility. A cron job would have silently failed.
Why MySQL over a time-series DB? At 500K events, MySQL with proper timestamp indexing was more than sufficient. I didn't want to add operational complexity without a real performance need.
The 70% correlation wasn't a hypothesis — I ran lag-correlation across event types and the 2–3 day signal kept appearing. That only shows up when you actually look at the data.
what I'd do differently

Add a proper alerting layer and explore whether an LSTM on the lag features improves prediction beyond correlation analysis. That's on the roadmap.

GitHub Demo
// how I approached this
the problem

The hard part wasn't the statistics — rolling averages are well-understood. The challenge was making them work in real-time at volume. At 5,000 updates/min across 50 symbols, naive implementations fall apart. This was really an exercise in thinking about data structures first.

key decisions
O(1) via running aggregates — I maintain hash maps keyed by symbol and update running state on each tick rather than reprocessing history. When a new update arrives, it's a constant-time operation.
Dash over Streamlit here — this dashboard needed smooth real-time updates without full page refreshes. Dash's callback architecture handles live data better for this use case.
what I'd do differently

Add a message queue (Kafka or Redis Pub/Sub) between ingestion and analytics. Right now they're tightly coupled — decoupling them would make the system more resilient and easier to scale.

Feb 2025 – Apr 2025
YouTube Virality Prediction 🎬
10,000+ videos across 5 countries via YouTube API — feature engineering on temporal patterns, engagement ratios, and metadata.
Logistic regression classifier — 70% accuracy, validated with train-test split and cross-validation. No overfitting shortcuts.
Findings via interactive Tableau dashboard. Virality is less random than it looks.
PythonPandasNumPy Scikit-learnYouTube APITableau
GitHub Dashboard
// how I approached this
the problem

Most answers to "what makes a video go viral?" are vague. I wanted an empirical approach — collect enough data to test actual hypotheses. The multi-country scope was intentional: I wanted to see whether virality signals were platform-universal or culturally specific.

key decisions
Why logistic regression over a complex model? Interpretability. A random forest might have hit 73% instead of 70%, but I couldn't tell you which features drove virality. With logistic regression, the coefficients are the answer — and that's what's actually interesting.
Feature engineering was the real work — raw counts (views, likes) are too correlated with video age to be useful. I engineered ratios, temporal features (time-to-trending), and metadata signals to get at the actual predictive signal.
Some features were universal, others weren't — engagement ratios predicted virality across all 5 countries; optimal upload timing varied significantly. That nuance disappears if you only look at one market.
what I'd do differently

Add a time-series component tracking a video's trajectory in the first 6 hours. The current model treats each video as a static snapshot. Early dynamics are probably more predictive than the final state.
